# Feature: buff-er

> Generated by `/feature` on 2026-03-28. Review before implementation.

---

## Intent

### 1. What problem does this solve?
Developers don't take physical breaks during the workday, contributing to burnout and sedentary health issues. Natural break points exist during AI wait times but nothing prompts physical movement.

### 2. How will we know it worked?
**Primary metric:** Hook retention rate over time — are users keeping it enabled after 2+ weeks?
**Secondary metric:** Self-reported exercise completion rate via follow-up prompt ("did you do it?").

### 3. What is out of scope?
- Personalized exercise plans or progression tracking
- Health/symptom management
- Cumulative stats ("you've done 50 pushups this week") — not in v1, future consideration
- Health app / wearable integration — not in v1
- Advanced fitness tracking or workout history

**In scope:**
- Suggest exercise during long AI waits
- Ask if they did it (simple yes/no tracking)
- Let users customize which exercises get suggested

### 4. What must NOT happen?
- Must not block or delay the AI operation — the hook is passive, never pauses work
- Must not suggest exercises over ~15 minutes
- Must not fire on short tasks — erodes trust and kills retention
- Must not crash or break the AI workflow on any failure — always exit 0, surface errors via `systemMessage`

### 5. Pre-mortem findings
**User-identified risks:**
- People don't like to exercise, so they ignore or uninstall (motivation problem, not a tool problem)
- Suggestions too hard — mitigated by customization
- Suggest too often — needs throttling/cooldown logic

**Additional risks identified:**
- Estimation accuracy is bad early on — fastest path to uninstall. Mitigated by learning from actual data (silent learning phase)
- Suggestions feel robotic/repetitive — need variety in the exercise catalog
- Follow-up "did you do it?" becomes annoying guilt trip — handle carefully, don't nag
- Wrong context (meeting, coffee shop) — subset of "too frequent" problem, covered by cooldowns

### Thinking Record: Intent
**Decided:** Build a Claude Code hook that suggests short exercises during long AI wait times, with a secondary time-since-last-break trigger. The real success metric is retention (and stars/installs for visibility), not exercise compliance.
**Reasoning:** The problem is sedentary developer behavior. AI wait times are a natural intervention point. The tool needs to be delightful enough to keep installed, not annoying enough to disable. Marketing/visibility value (stars, installs) is an explicit success criterion alongside genuine user benefit.
**Considered and rejected:** Treating this as a "dead time" problem rather than a "missed exercise opportunity" problem — reframed to focus on the real issue (lack of physical breaks, not idle time).
**Data consulted:** Claude Code hooks documentation — confirmed hook lifecycle, available events, data payloads, and constraints.
**Assumptions:** Users are willing to exercise if nudged at the right moment; AI wait time estimation is possible with reasonable accuracy via learned timing data; short exercises (1-5 min) are valuable enough to matter.
**Risks accepted:** Most users will ignore suggestions (acceptable — retention and visibility matter more than compliance). Estimation will be imperfect at launch (learning phase mitigates).
**Skipped/declined:** None.

## Behavioral Spec

### User Stories

**Today:** AI task starts -> developer stares at terminal or drifts to Slack/coffee -> AI finishes -> developer context-switches back. No physical movement, no prompt to move.

**Desired:** AI task starts -> hook estimates duration from learned timing data -> if long enough, suggests a specific exercise matched to the estimated wait time -> developer does the exercise -> AI finishes -> hook asks "did you do it?" -> developer answers -> back to work.

**Secondary flow (time-since-break):** Developer has been working for 45+ minutes with many short AI tasks -> none triggered the command estimator -> at the next `Stop` event, hook suggests a break because it's been too long since the last one.

### Mechanism

The tool has two trigger modes that coexist:

**Trigger 1 — Learned command estimation:**
- `PostToolUse` hook records actual duration for every Bash command, keyed by command pattern (first two tokens, e.g., `cargo build`)
- Builds per-project timing database with 2-3 day sliding window
- Once 3+ samples exist for a pattern: check average duration and 75th percentile
- If estimated time > threshold (configurable, default 3-5 min): fire exercise suggestion on `PreToolUse` before the command runs
- Silent during learning phase (first 3 runs of any command)

**Trigger 2 — Time since last break:**
- Track when the last exercise suggestion was shown
- At `Stop` events, check elapsed time since last suggestion
- If over cooldown threshold (configurable, default ~30-45 min), suggest exercise at the next natural pause

**Why this works:** Learning from actual data avoids bad heuristics. Per-project timing accounts for different build times across repos. The time-since-break trigger catches long sessions of many short tasks. Both triggers respect cooldowns to avoid nagging.

### States and Transitions

1. **Idle** -> hook triggers on AI command
2. **Estimating** -> evaluate command pattern against timing database
3. **Decision gate** -> if no data (learning phase) or estimated time < threshold, go to Idle silently. If estimated time >= threshold, continue
4. **Exercise suggestion** -> match remaining time to an appropriate exercise from catalog, emit suggestion via `systemMessage`
5. **Waiting** -> AI task running, user is free
6. **Complete** -> task finishes, "did you do it?" prompt fires immediately via next hook event
7. **Idle** -> user responds (or doesn't), back to waiting

Time-since-break trigger:
1. **Stop event fires** -> check time since last suggestion
2. **Over threshold** -> suggest exercise
3. **Under threshold** -> do nothing

### Error Cases

| Failure | Behavior |
|---------|----------|
| Timing DB corrupted | Log error, surface via `systemMessage` ("run `buff-er doctor`"), skip suggestion, exit 0 |
| Config parse error | Log error, surface via `systemMessage`, fall back to defaults, exit 0 |
| Disk full / write fails | Skip recording, skip suggestion, exit 0 |
| Exercise catalog empty (bad custom config) | Log warning via `systemMessage`, skip suggestion, exit 0 |
| Concurrent sessions writing to DB | SQLite handles this natively |
| Estimation wildly wrong | Accept the risk, track accuracy metrics for future refinement |
| Hook crashes for any reason | Must exit 0, never break AI workflow |
| Rapid successive tasks | May show two suggestions — acceptable risk |

### Thinking Record: Behavioral Spec
**Decided:** Two coexisting trigger modes (learned command estimation + time-since-break). Learning-based approach with 3-sample minimum, average + 75th percentile check, 2-3 day sliding window. All failures exit 0 with user-visible error messages via `systemMessage`.
**Reasoning:** Learning from actual data is more accurate than static heuristics and adapts to project/machine. Time-since-break catches long sessions that command estimation misses. Silent learning phase avoids bad early suggestions that kill trust.
**Considered and rejected:** Static command heuristics (too inaccurate), transcript parsing for plan estimation (fragile/unreliable), standalone daemon monitoring idle gaps (detects existing breaks rather than creating them — wrong direction), firing suggestion after task completes (wrong timing — user needs the nudge before the wait, not after).
**Assumptions challenged:** "Can we see the AI's planned tool queue?" — No, hooks fire one tool at a time with no planning visibility. This forced the learning-based approach over plan-based estimation.
**Risks accepted:** Silent during learning phase. Double-firing on rapid successive tasks. Estimation accuracy limited to command-level patterns.

## Design Approach

### Conceptual approach
Claude Code hook implemented as a Go single binary. Hooks into `PreToolUse` (check estimates, suggest exercises), `PostToolUse` (record actual durations), and `Stop` (time-since-break check). Learns command timing per project, stores data in XDG-standard paths, ships with a default exercise catalog that users can customize.

Permission batching (front-loading AI approvals so users can walk away) is a separate but related feature — architecturally decoupled, built alongside, can be extracted later.

### Codebase context
Greenfield project — no existing code. Claude Code hooks are configured in `~/.claude/settings.json` and receive JSON on stdin, return JSON on stdout. Hook events: `PreToolUse`, `PostToolUse`, `Stop` are the relevant ones. The binary will be invoked by Claude Code as a subprocess on each hook event.

### Thinking Record: Design Approach
**Decided:** Go single binary, Claude Code hook integration, learned timing estimation, XDG storage.
**Reasoning:** Go chosen over Rust for easier AI-assisted development and review — simpler language means bugs are more visible. Single binary for zero-dependency install. Learning-based estimation because hooks provide no planning visibility. XDG paths follow Unix conventions.
**Alternatives rejected:** Rust (harder for AI to vet), Python/Node (runtime dependencies), standalone daemon (wrong interaction model), static heuristics (inaccurate).
**Codebase findings:** Empty repo. Claude Code hooks documented — PreToolUse/PostToolUse provide tool name and input, Stop provides last message. No queue visibility.
**Tradeoffs accepted:** Go's verbosity vs Rust's type safety. Silent learning phase means no value on first install until data accumulates. Command grouping by first two tokens may be too coarse for some commands.

## Implementation Design

### Architecture

```
buff-er/
├── cmd/
│   └── buff-er/
│       └── main.go            # CLI entrypoint, subcommands (hook, install, doctor, config)
├── internal/
│   ├── hook/
│   │   ├── handler.go         # Hook event dispatch (PreToolUse, PostToolUse, Stop)
│   │   └── input.go           # JSON payload parsing from stdin
│   ├── timing/
│   │   ├── store.go           # SQLite read/write at XDG data path
│   │   ├── estimator.go       # Average, 75th percentile, sliding window logic
│   │   └── command.go         # Command pattern extraction (first 2 tokens)
│   ├── exercise/
│   │   ├── suggest.go         # Pick exercise based on estimated duration
│   │   └── catalog.go         # Default catalog + user override merge
│   └── config/
│       └── config.go          # User config loading, validation, fallback to defaults
├── go.mod
├── go.sum
└── install.sh                  # Hook registration into Claude Code settings (merge, not overwrite)
```

**Storage paths:**
- Timing data: `~/.local/share/buff-er/{project-hash}/timings.db` (SQLite)
- User config: `~/.config/buff-er/config.toml`
- Logs: `~/.local/share/buff-er/logs/`

**Data model:**
- `TimingRecord`: command_pattern (string), start_time (timestamp), duration_ms (int), project_hash (string)
- `ExerciseEntry`: name (string), description (string), min_duration_min (int), max_duration_min (int), category (string: stretch/strength/cardio)
- `UserConfig`: custom_exercises ([]ExerciseEntry), min_trigger_minutes (float), break_cooldown_minutes (int), enabled (bool)

**Integration points:**
- Claude Code invokes `buff-er hook` as a subprocess, passing JSON on stdin
- `buff-er install` merges hook config into `~/.claude/settings.json`
- `buff-er doctor` checks DB health, config validity, hook registration
- `buff-er config` manages exercise list and thresholds

**Hook registration (what `install.sh` writes to settings.json):**
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{"type": "command", "command": "buff-er hook pre-tool-use", "timeout": 5}]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{"type": "command", "command": "buff-er hook post-tool-use", "timeout": 5}]
      }
    ],
    "Stop": [
      {
        "hooks": [{"type": "command", "command": "buff-er hook stop", "timeout": 5}]
      }
    ]
  }
}
```

### Structural issues flagged
Greenfield — no existing slop to clean up. The main structural risk is `install.sh` modifying `~/.claude/settings.json` — this must merge into existing hooks, not overwrite them. This is where bugs will hide.

### Constraints and Non-negotiables
- **Security:** No sensitive data handled. Timing data and exercise lists aren't secrets. No network calls. No auth.
- **Reliability/SLA:** Every failure mode exits 0. Hook timeout set to 5 seconds — must respond faster than that. SQLite queries on small dataset should be <10ms.
- **Compliance/policy:** None applicable.

### Known AI Blind Spots
- [x] Edge cases: Corrupted DB, empty config, disk full — all handled by "exit 0, surface via systemMessage" pattern. Parse failures in config fall back to defaults.
- [x] Concurrency: Multiple Claude Code sessions writing timing data simultaneously — SQLite handles this natively with WAL mode.
- [x] Error handling: Every code path must exit 0. Use `systemMessage` to surface issues to users. Never panic, never exit non-zero (except exit 2 for intentional blocks, which this tool doesn't use).
- [x] Security: Low risk — no secrets, no network, no auth. Exercise suggestions are not sensitive data.
- [x] Non-prompted concerns: `install.sh` must merge into existing settings.json hooks, not overwrite. Sliding window cleanup needs to run periodically (on each write, prune records older than 2-3 days). Log rotation if logging to files.
- [x] Hallucination risk: Verify Go SQLite packages exist (e.g., `modernc.org/sqlite` for pure-Go, no CGO). Verify XDG path handling libraries. AI tends to invent Go package names — all imports must be verified before build.

### Rollback Plan
- **Trigger signal:** Users report hook errors, or `systemMessage` errors appearing frequently in own usage
- **Method:** `buff-er uninstall` removes hook entries from Claude Code settings. Binary on disk does nothing without hook registration. Can also disable via config (`enabled: false`).
